# 기능 배포 (이슈 → 브랜치 → TDD → lint/test → 커밋 → 푸시 → PR)

이 프로젝트 `CLAUDE.md`에 정의된 로컬 작업 방식을 자동화한다: 요청받은 함수/기능을
추적용 GitHub 이슈로 등록하고, TDD로 구현하고, 이 저장소의 검사 기준으로 검증한 뒤
그 이슈를 닫는 PR까지 연다 — 매번 사용자가 git/이슈 단계를 하나하나 지시하지 않아도
되도록 한다.

이 워크플로우는 전부 **지금 이 로컬 세션 안에서** 진행된다. `CLAUDE.md`의 "클라우드"
방식(이슈를 만들면 GitHub Actions가 대신 작업하는 방식)과는 다르다 — 여기서는 이슈를
작업 기록/추적용으로만 만들고, 실제 구현·커밋·푸시·PR은 전부 이 세션의 Claude가
직접 수행한다.

## 입력

사용자가 설명하는 기능 내용: 대상 파일, 함수 시그니처, 기대 동작, 예시 케이스.
이 중 빠졌거나 애매한 부분(예: 예시 케이스가 없거나 엣지 케이스 동작이 불명확한 경우)이
있으면 코드를 쓰기 전에 짧게 확인 질문을 한다 — 추측한 명세로 작성한 테스트는
테스트가 없느니만 못하다.

## 진행 순서

1. **GitHub 이슈 생성.** 사용자가 설명한 기능 내용을 정리해 이슈로 등록한다:
   ```
   gh issue create --title "<간결한 제목>" --body "<기능 설명 + 기대 동작 + 예시 케이스>"
   ```
   출력된 이슈 번호(`#<n>`)를 이후 단계(브랜치명, 커밋, PR)에서 계속 사용한다. 이 이슈는
   작업이 뭘, 왜 하는지에 대한 기록이지 실행 주체가 아니다 — 실제 작업은 이어서 이
   세션이 직접 한다.

2. **저장소 상태 확인.** `git status`를 실행한다. 이 작업과 무관한 커밋되지 않은
   변경사항이 있으면, 새 브랜치에 섞지 말고 먼저 사용자에게 어떻게 처리할지 물어본다.
   이미 이 작업에 맞는 브랜치에 있다면 그대로 사용하고, 아니면 `main`에서 새로
   분기한다. 브랜치명에는 이슈 번호를 넣어 이 저장소의 기존 관례(`issue-3-...`)를
   따른다:
   ```
   git checkout main
   git pull
   git checkout -b issue-<n>-<짧은-kebab-slug>
   ```
   `main`에는 절대 직접 커밋하거나 푸시하지 않는다 (`CLAUDE.md` 참고).

3. **실패하는 테스트를 먼저 작성 (TDD).** `CLAUDE.md`에 따라 구현보다 테스트가 먼저다.
   `pytest`를 사용해 사용자가 설명한 모든 케이스(일반적인 입력, 빈 입력 같은 엣지
   케이스, 대소문자/포맷 변형 등)와 사용자가 언급하지 않았지만 명세상 당연히 필요한
   엣지 케이스까지 커버한다. 새 테스트가 실제로 실패하는지 한 번 `pytest`를 돌려
   확인한다 (테스트가 진짜로 뭔가를 검증하고 있다는 증거이며, 항상 통과하는 무의미한
   테스트가 아님을 보장한다).

4. **구현.** 테스트를 통과시키는 데 필요한 최소한의 코드를, 사용자가 지정한 파일(없으면
   명백히 대응되는 모듈)에 작성한다.

5. **검증.** 이 프로젝트의 표준 검사를 실행하고 둘 다 통과할 때까지 반복한다:
   ```
   ruff check .
   pytest
   ```
   `./scripts/check.sh`가 있으면 그것을 우선 사용한다 — CI와 동일한 검사를 그대로
   재현한다. 둘 중 하나라도 실패하는 상태에서는 커밋하지 않는다; 검사를 우회하거나
   건너뛰지 말고 원인을 고친다.

6. **커밋.** 이 변경과 관련된 파일만 스테이징한다 (`git add -A`를 무작정 쓰지 말고
   먼저 `git status`로 확인한다). 무엇을, 왜 바꿨는지 한 줄로 명확히 쓴 커밋 메시지를
   작성한다 (`CLAUDE.md` 규칙).

7. **푸시하고 PR 오픈:**
   ```
   git push -u origin <브랜치>
   gh pr create --base main --title "<간결한 제목>" --body "<요약 + 테스트 계획>

Closes #<n>"
   ```
   `Closes #<n>`을 반드시 PR 본문에 포함해 머지 시 1단계에서 만든 이슈가 자동으로
   닫히도록 한다 (`CLAUDE.md` 커밋/PR 규칙). PR 본문에는 무엇을 추가했는지, 어떤
   테스트 케이스를 커버했는지 간단히 적는다 — 리뷰어가 훑어보고 머지할 수 있을
   정도면 충분하다.

8. **결과 보고.** 이슈 URL, 브랜치 이름, PR URL을 모두 사용자에게 전달한다.
   "완료했습니다" 정도로 끝내지 말 것 — 이슈/PR 링크가 사용자가 실제로 필요한
   결과물이다.

## 참고

- 이 워크플로우는 소규모·저위험 저장소(Python 3.12, pytest, ruff — 전체 규칙은
  `CLAUDE.md` 참고)를 전제로 한다. 한 번의 TDD 사이클로 끝낼 수 있을 만큼 범위가
  좁은 변경을 가정하므로, 실제 요청이 여러 부분으로 이루어진 큰 기능이라면 이 흐름에
  억지로 욱여넣지 말고 더 작은 PR들로 나눌 것을 먼저 제안한다.
- `ruff check .` 또는 `pytest`가 몇 번 고쳐도 계속 실패하면, 우회하지 말고(예: lint
  규칙을 무시하거나 테스트를 지워서 통과시키는 식) 실제 에러를 그대로 사용자에게
  보여주고 멈춘다.
- 이슈 생성, 푸시, PR 오픈은 모두 GitHub에 남는 눈에 보이는 작업이지만, 이 저장소에서는
  기능 추가 요청 자체가 곧 "끝까지 배포해달라"는 뜻으로 이미 확립된 관례다. 매번
  "이슈 만들고 커밋/푸시/PR까지 해줘"라고 되풀이해서 말할 필요 없이, 함수/기능
  추가 요청이면 바로 1단계(이슈 생성)부터 시작한다. 사용자가 명시적으로 "일단
  코드만 보여줘", "커밋은 하지 마" 같이 반대로 말한 경우에만 이 기본값에서 벗어난다.
